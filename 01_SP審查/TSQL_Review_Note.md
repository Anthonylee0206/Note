# T-SQL 腳本審查筆記

> SQL Server 2022 ｜ 協助執行 RD 腳本前的自我檢查參考

---

## 一、執行安全性

| 檢查項目 | 說明 |
|---------|------|
| 確認目標資料庫 | 腳本開頭是否有 USE [DatabaseName]，確認不會跑錯庫 |
| 確認執行環境 | 開發 / 測試 / 正式環境，是否符合預期 |
| 確認是否有最新備份 | 正式環境執行前，確認已有可用備份 |
| 確認回滾方案 | DML 操作應提供回滾腳本，或包在 BEGIN TRAN 中先確認再 COMMIT |
| 檢查破壞性操作 | 注意 DROP、TRUNCATE、DELETE 無 WHERE 等危險語法 |
| 確認執行時段 | 高負載或破壞性操作安排離峰時段執行 |

## 二、影響範圍評估

| 檢查項目 | 說明 |
|---------|------|
| 先用 SELECT 確認影響筆數 | UPDATE / DELETE 先改寫成 SELECT COUNT(*) 確認範圍，幾百筆和幾百萬筆處理方式不同 |
| 大量資料需分批處理 | 影響超過萬筆以上，用 UPDATE TOP (N) + WHILE 迴圈分批，避免長時間鎖定與 Log 暴漲 |
| 注意交易日誌 | 大量 DML 操作可能導致 Transaction Log 暴漲，確認磁碟空間足夠 |

## 三、鎖定與阻塞風險

| 檢查項目 | 說明 |
|---------|------|
| 評估鎖定範圍 | 大範圍 UPDATE / DELETE 可能觸發 Lock Escalation 升級至表鎖，影響線上服務 |
| 注意 Schema Lock | ALTER TABLE 等 DDL 會持有 Schema Lock，繁忙時段會阻塞所有查詢 |
| 查詢是否加 WITH (NOLOCK) | SELECT 視情況加 NOLOCK 提示，避免不必要的共用鎖定 |

## 四、WHERE 條件

| 檢查項目 | 說明 |
|---------|------|
| 確認有 WHERE 條件 | UPDATE / DELETE 必須有 WHERE，避免誤改全表 |
| WHERE 條件是否合理 | 條件能否精確定位目標資料，不會多刪多改 |

## 五、資料型態與隱含轉換

| 檢查項目 | 說明 |
|---------|------|
| 參數型態與欄位一致 | 例如欄位 VARCHAR 但參數用 NVARCHAR，會導致整欄隱含轉換（Implicit Conversion），索引失效變成全表掃描 |
| JOIN 條件型態一致 | 兩表 JOIN 欄位型態不一致同樣造成隱含轉換 |
| 檢查 CONVERT_IMPLICIT | 執行計畫中出現黃色驚嘆號或 CONVERT_IMPLICIT 警告，需修正型態 |

## 六、索引與 SARGability

| 檢查項目 | 說明 |
|---------|------|
| 不要對欄位做函數運算 | `WHERE YEAR(Col) = 2025` 改為 `WHERE Col >= '2025-01-01' AND Col < '2026-01-01'` |
| 運算放在值那邊 | `WHERE Price * 1.05 > 100` 改為 `WHERE Price > 100 / 1.05`，保持欄位乾淨讓索引可用 |
| 避免 SELECT * | 明確指定需要的欄位，減少 I/O 與網路傳輸 |
| 檢查 Table Scan | 用「顯示估計執行計畫」確認大表上有沒有不預期的全表掃描 |

## 七、寫法與效能

| 檢查項目 | 說明 |
|---------|------|
| 避免 Cursor | 大部分情境可用集合式操作取代 |
| 避免迴圈逐筆操作 | 應盡量改為批次集合操作 |
| 暫存表 vs 資料表變數 | 大量資料用暫存表 (#temp)，少量用資料表變數 (@table) |
| 子查詢考慮改 JOIN | 相關子查詢（Correlated Subquery）改為 JOIN 通常效能更好 |

## 八、安全性與權限

| 檢查項目 | 說明 |
|---------|------|
| 動態 SQL 風險 | `EXEC(@sql)` 有 SQL Injection 風險，建議用 `sp_executesql` 參數化 |
| xp_cmdshell | 確認是否必要，避免不當的系統層級操作 |
| 執行權限最小化 | 不使用不必要的高權限帳號 |
